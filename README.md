Memo-to-Wiki Multi-Agent System
-사용자가 일상적으로 기록하는 파편화된 메모들을 분석하여 유사한 주제끼리 자동 분류하고, 유기적인 위키(Wiki) 페이지로 정돈해 주는 멀티 에이전트 서비스입니다
-본 프로젝트는 메모를 개별적으로 열어봐야 하는 번거로움을 해결하고, 파편화된 지식을 하나의 유기적인 지식 구조(Knowledge Base)로 연결하기 위해 기획되었습니다.

System Architecture & My Role
-전체 시스템은 1차 데이터 가공 및 정제 에이전트와 2차 임베딩 및 클러스터링 에이전트의 파이프라인으로 구성되어 있습니다.저는 이 중 1차 가공 에이전트의 설계 및 구축 전체를 전담했습니다.

Key Achievements & Troubleshooting (본인 기여도 100%)

1. 통계학적 관점을 통한 데이터 정의 정립 (기획 병목 해결)

Problem: 

-개발 초기 단계에서 팀원 간 '메모'와 '정리'라는 모호한 개념에 대한 합의가 부족해 아키텍처 설계가 지연되었습니다.

Action: 

-통계학 전공 경험을 바탕으로, 변수의 엄밀한 정의가 최종 분석 모델의 신뢰도에 미치는 중요성을 제기했습니다. 회의를 주도하여 입력 데이터 유형(공부 내용, 회의록, 플래너, 일상 기록 등)과 길이별 속성을 구조화하여 카테고리를 명확히 정의했습니다.

Result: 

-개발 가이드라인과 기준을 조기에 확립하여 파이프라인 설계를 속도감 있게 추진할 수 있었습니다.

2. LangGraph 기반 동적 라우팅 및 멀티 포맷 파이프라인 구축

Problem: 

-단일 형태의 전처리 파이프라인으로는 텍스트, PDF, Word 등 각기 다른 원천 문서 형식과 다양하게 흩어진 메모 유형들을 수용하기 어려웠습니다.

Action:

-조건부 분기(Conditional Branching)와 워크플로우 제어가 유연한 LangGraph를 활용하여 데이터 제어 흐름을 설계했습니다.

-입력 흐름 최전방에 포맷 변환 전처리 노드를 추가하여 오케스트레이션 방식으로 통합했습니다.

-분류된 문서 성격에 맞춰 적절한 템플릿과 정제 필터가 적용되도록 Fan-out/Fan-in 구조의 동적 라우팅 노드를 개발했습니다.

Result: 
-어떤 형식의 메모 파일이 업로드되어도 누락 없이 안정적으로 정제 데이터를 전달할 수 있는 단일 인풋 파이프라인을 완성했습니다.

3. 하드웨어/비용 제약을 고려한 하이브리드 LLM 아키텍처 설계

Problem: 

-아이패드 손글씨 이미지(OCR) 처리를 위해 멀티모달(LMM) 도입이 필요했으나, 로컬 GPU 환경에서 8B 급 대형 모델 구동 시 1~2분의 심각한 지연이 발생해 상용 UX 기준에 부적합했습니다. 고사양 클라우드(AWS EC2) 도입은 제한된 예산을 크게 초과하는 문제가 있었습니다.

Action:

-경량 멀티모달 모델인 Kanana-1.5-v-3b의 로컬 배포를 직접 실행하여 하드웨어 자원 점유율과 대기 시간 간의 Trade-off를 실험 및 검증했습니다.

비용, 가공 품질, 처리 속도의 세 가지 차원을 수치화하여 비교 분석한 끝에, GPT-4o-mini 및 GPT-4o 하이브리드 API 호출 구조를 제안 및 도입했습니다.

Result: 
-성능 품질을 상용 서비스 수준인 90% 이상으로 유지하면서도 예산 범위 내로 비용을 방어하고, 응답 지연을 대폭 단축하여 극적인 최적화를 이뤄냈습니다.
